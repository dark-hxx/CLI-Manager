# CLI-Manager Web 产品范围与实施计划

## 1. 目标

Web 端是桌面端的远程控制、结构化协作和数据查看界面。桌面端是全部业务数据和执行结果的唯一来源，继续持有项目文件、CLI、PTY、Git、凭据及系统资源；Web 端只发送操作请求、接收桌面上报并提供缓存，不直接执行业务操作，也不复刻终端。

## 2. 能力边界

| 类型 | 说明 | 典型能力 |
|---|---|---|
| 桌面执行 | Web 发请求，桌面执行并上报结果 | AI 对话、文件、Git、Worktree、备份 |
| 桌面上报 | Web 展示桌面产生的结构化数据 | 历史、分析、请求日志、设备状态 |
| Web 缓存 | 缓存最近一次桌面上报，仅用于加速和离线只读 | 项目索引、会话、统计、Diff 摘要 |
| 桌面专属 | Web 只提供配置说明或跳转 | PTY、终端主题、桌宠、托盘、外部终端 |

Web 缓存不是独立数据源，不允许在桌面未确认时生成业务事实、推断操作成功或覆盖桌面状态。

### 2.1 Web 技术栈

- 后端：Rust 模块化单体，同时提供 API、WebSocket 网关和业务逻辑。
- 数据库：SQLite3，保存关系数据、审计和桌面上报缓存。
- 缓存：首版使用进程内缓存和 SQLite3；Redis 默认不引入，仅在架构升级为多实例且存在明确跨实例需求时重新评估。
- 移动端：响应式 Web/PWA，不单独开发原生 App。

## 3. MVP 优先级

| 阶段 | 范围 | 验收结果 |
|---|---|---|
| P0 | 登录、设备配对、工作台、结构化对话、审批、历史 | 能安全完成远程 AI 任务闭环 |
| P1 | 项目、Worktree、文件、Diff、Git | 能查看和批准代码变更 |
| P2 | Replay、Prompt、分析、请求日志、设备任务 | 能审计过程并分析长期使用数据 |
| P3 | 供应商、Hook、备份、设置、移动端完善 | 完成桌面端管理能力映射 |

## 4. 已确认的 UI 方向

Web 端采用浅色、深色两套主题。浅色主页面以已确认的桌面端和移动端原型为基准，后续页面必须沿用相同布局、组件和视觉语言，不再重新设计第三套风格。

设计参考：[Stack-Cairn/LiveAgent](https://github.com/Stack-Cairn/LiveAgent)。参考范围为 Gateway WebUI 的页面组织、对话空状态、历史侧栏、右侧工具 Dock、主题 Token 和移动端响应式方式；不复制 LiveAgent 名称、Logo、文案、业务模型或具体代码。

### 4.1 浅色主题

- 页面背景使用淡蓝灰，主要内容使用白色表面。
- 品牌和主操作使用蓝色；成功、提醒和风险分别使用低饱和绿色、橙色和红色。
- 卡片使用细边框、极轻阴影和充足留白，不使用玻璃拟态或大面积渐变。
- 主页面强调“立即开始任务”，不做传统数据看板式首页。
- 桌面端采用左侧导航、中间工作区、右侧快捷任务三栏结构。
- 移动端采用单栏内容、固定输入区和底部导航。

### 4.2 深色主题

- 保留现有 terminal-green 深色原型。
- 与浅色主题共享信息架构、组件尺寸、页面层级和交互流程。
- 仅切换颜色、阴影、代码主题和图表配色，不产生两套功能逻辑。

### 4.3 主页面固定结构

| 区域 | 桌面端 | 移动端 |
|---|---|---|
| 顶部 | 项目、分支、设备、主题和布局控制 | 项目、模型、在线状态和菜单 |
| 导航 | Skills、MCP、新对话、工作空间、最近会话 | 底部工作台、项目、历史、分析、我的 |
| 主区 | 问候语、快捷 Prompt、大输入框 | 问候语、纵向快捷卡片、最近会话 |
| 辅助区 | 新建对话、打开项目、代码审查、SSH、后台任务 | 待审批摘要和抽屉入口 |
| 输入区 | 中间区域底部固定 | 底部导航上方固定 |

### 4.4 从 LiveAgent 吸收的设计模式

| LiveAgent 设计模式 | CLI-Manager Web 调整方式 |
|---|---|
| 对话是应用主工作区 | Web 首页直接进入结构化 AI 对话，不先进入统计看板 |
| 左侧统一承载项目和历史 | 左侧保留 Skills、MCP、新对话、工作空间和最近会话 |
| 空状态按时间显示问候语 | 使用早上好/下午好等动态问候，并提供任务建议卡 |
| 建议卡直接填充 Prompt | “解决项目、修复问题、头脑风暴”点击后生成可编辑 Prompt，不立即发送 |
| 输入框支持引用 | `@` 引用项目文件，`$` 引用 Skills，附件使用独立入口 |
| 模型和推理控制靠近输入区 | 模型、推理等级、权限策略放在输入框底部工具栏 |
| 右侧 Dock 按需打开工具 | 右侧承载文件、Git、Diff、SSH、后台任务和审批，不长期占满主区 |
| 历史切换保持对话上下文 | 切换会话时保留草稿、滚动位置和未完成上传 |
| 浅色/深色由 Token 驱动 | 两套主题共享组件和布局，只切换 CSS 语义 Token |
| Gateway 断线可恢复 | 展示连接状态、事件补传和最后同步时间，不直接清空当前页面 |

### 4.5 明确不照搬的内容

- Web 主工作区不提供 xterm 终端；命令执行继续显示为结构化工具卡和审批。
- 不复用 LiveAgent 的品牌、鸟形图标、OpenAI 字体资源和产品文案。
- 不直接照搬其多模型供应商、记忆、自动化或隧道业务结构。
- 不让文件树、Git、SSH 等工具默认同时展开；通过右侧 Dock 按任务需要打开。
- 不因参考项目已有实现而改变 CLI-Manager“桌面端执行、Web 端控制和查看”的数据边界。

## 5. 实施工作包

1. 建立设备身份、用户身份、会话身份和项目身份映射。
2. 定义桌面上报快照、增量事件、心跳、命令回执和缓存版本协议。
3. 建立 Web 只读缓存层，支持首次加载、断线查看、增量更新、过期标记和主动失效。
4. 建立语义化主题 Token，先完成浅色/深色切换，再实现全局三栏和移动单栏框架。
5. 实现历史/工作空间侧栏，包含搜索、运行状态、最近会话、草稿和移动端抽屉。
6. 实现主页面空状态：动态问候、建议卡、`@`/`$` 引用输入和模型/推理控制。
7. 实现对话任务流、流式消息、执行时间线、审批、文件变更和连接恢复状态。
8. 实现右侧工具 Dock，按需承载文件、Git、Diff、SSH、后台任务和设备状态。
9. 接入桌面上报的历史、Replay、Prompt、分析、请求日志、设备和管理数据。
10. 完成离线、超时、冲突、重连、事件补传、重复提交和撤销策略。
11. 完成响应式、键盘操作、屏幕阅读器、减少动画和中英文适配。
12. 建立桌面协议 Mock Server 和缓存快照测试，使移动 Web 开发不依赖真实桌面在线。

## 6. 页面实施顺序

| 顺序 | 页面 | 关键验收 |
|---:|---|---|
| 1 | 全局框架与主页面 | 桌面、移动端与确认原型一致 |
| 2 | 历史侧栏与输入系统 | 草稿、会话切换、`@`/`$` 引用可靠 |
| 3 | 对话执行与审批 | 空状态进入任务后布局平滑切换 |
| 4 | 右侧工具 Dock | 文件、Git、Diff、SSH、后台任务按需切换 |
| 5 | 项目与 Worktree | 延续同一导航、卡片和状态层级 |
| 6 | 历史、Replay、Prompt | 高密度信息仍保持浅色主题可读性 |
| 7 | 分析、请求日志、设备 | 图表和表格使用双主题专用配色 |
| 8 | 供应商、Hook、备份、设置 | 表单、危险区和权限提示统一 |

## 7. 非目标

- 浏览器内运行真实 PTY 或模拟 ANSI 终端。
- 把桌面凭据、供应商密钥或 WebDAV 密码传入浏览器。
- Web 端绕过桌面代理直接访问用户文件系统。
- Web 服务独立生成项目、会话、Git、文件、统计或设备业务数据。
- 设备离线时排队执行文件写入、Git、Worktree、Hook、备份等操作。
- 首版支持多人同时编辑同一个工作区。

## 8. 交付标准

- 每项功能明确数据来源、权限、状态和异常。
- 所有写操作显示目标设备、项目、Worktree 和影响范围。
- 危险操作支持二次确认，审批记录可追溯。
- 设备离线时界面不展示虚假的可执行状态。
- 桌面、平板和移动端均无整页横向滚动。
- 用户可见文案同时定义 `zh-CN` 与 `en-US`。
- 浅色页面必须与确认的主页面保持相同蓝色主色、白色表面、边框、阴影和留白体系。
- 深色与浅色主题必须功能一致，切换主题不得改变布局和操作位置。
- 代码、Diff 和日志允许使用独立高对比表面，但必须与当前主题边界清晰。
- 主题支持浅色、深色和跟随系统，并记忆当前浏览器偏好。
- 主页面、历史侧栏、输入框和右侧 Dock 的行为与 LiveAgent 参考模式一致，但业务字段和操作权限遵循 CLI-Manager 设计。
- 建议卡只生成可编辑 Prompt；任何桌面执行必须经过正式发送和既有权限检查。
- 断线重连后恢复当前会话、草稿和执行状态，不出现重复消息或重复任务。
- 每个页面能区分实时数据、缓存数据和过期数据，并显示最后桌面上报时间。
- 所有操作只有收到桌面回执后才能标记成功；Web 本地状态最多标记为“已提交”或“等待桌面”。
- Rust 网关和业务逻辑保持同进程、分模块，SQLite3 数据文件只由该服务实例管理，不提前拆成微服务。
- 移动 Web 冷启动优先展示最后有效缓存，并在桌面数据到达后原位刷新。
